# 接任务决策包 SOP v1

最后更新：2026-04-08

## 目标

让小青在接到新任务时，先完成目标收敛与制约判断，再决定是否执行或分发，减少失焦、返工和不良多工。

## 适用范围

适用于所有**任务相关请求**。

不适用于：
- 普通聊天
- 简单寒暄
- 无需持续跟踪的一次性超短问答

## 标准顺序

1. 先定义目标
2. 再判断当前主制约
3. 给出最佳实践方案
4. 收敛关键决策点
5. 决定执行路径
6. 再进入执行

## 标准输出模板

```md
当前任务：<TASK-ID> / <任务名> / <状态>
目标：<这次任务要达成什么>
当前主制约：<当前最可能卡住吞吐的点>
最佳实践方案：<默认推荐路径>
关键决策点：
1. <决策点1>
2. <决策点2>
执行路径：小青直做 / 分发给 <角色>
本次推进：<这条回复实际推进了什么>
```

## 使用规则

### 1. 目标
必须用一句话讲清楚交付结果，而不是动作。

错误：下载一张图
正确：补齐身份系统头像，使视觉身份完整可用

### 2. 当前主制约
只判断当前最主要的一个制约，不同时报多个。

### 3. 最佳实践方案
默认先给推荐方案，再给备选方案。
如果需要用户拍板，先把推荐路径说清楚。

### 4. 关键决策点
只保留真正需要俊阳拍板的 1 到 3 个点。
非关键点由小青默认决策。

### 5. 执行路径
- 简单、低风险、边界清晰：小青直做
- 架构、规划、编码、环境、调试、测试：按角色分发
- 如果当前主制约在主链路决策吞吐，优先先收敛信息，再分发

## 任务完成时追加

```md
本进程其他进行中：<列表；无则写 无>
本进程其他未完成：<列表；无则写 无>
```

## 试运行判定标准

一次试运行算成功，至少满足：
1. 回复中先出现目标、制约、最佳实践、决策点、执行路径
2. 没有过早进入执行细节
3. 如果任务完成，带出本进程其他未完成任务
4. 用户不需要额外追问“你现在到底在做什么”

## 失败信号

出现以下任一情况，视为 SOP 未执行好：
- 直接开做，没有先收敛目标
- 给了很多方案，但没有推荐路径
- 关键决策点没有压缩，导致用户需要手动整理
- 多任务切换时没有明确说明当前任务
- 任务完成后没有暴露其他未完成任务
