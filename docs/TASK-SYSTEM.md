# 任务系统 v1

最后更新：2026-04-08

## 1. 目标

在多任务并行环境下，保持焦点，避免不良多工，确保：
- 每次都知道当前正在推进哪项任务
- 任务完成时，能立即看到本进程还有什么没做完
- 普通聊天不被错误计入任务系统

## 2. 任务系统范围

### 计入任务的事项
- 明确要交付结果的工作
- 需要持续跟踪状态的工作
- 需要跨回复、跨子进程衔接的工作

### 不计入任务的事项
- 普通聊天
- 简单寒暄
- 一次性无需跟踪的超短问答

## 3. 单一事实来源

任务统一维护在：`TASKS.md`

`TASKS.md` 负责记录：
- 当前聚焦任务
- 进行中任务
- 等待 / 未完成任务
- 最近完成任务
- 按进程划分的任务视图

### 3.1 启动时待办加载协议

`TASKS.md` 同时承担 **待办启动快照** 的职责。

默认流程：
1. 会话启动时读取一次 `TASKS.md`
2. 提取 `最后更新`、`当前聚焦任务`、`进行中任务`、`等待 / 未完成任务`
3. 在当前会话内形成轻量 `TodoContext`
4. 后续高频待办查看直接使用 `TodoContext`，不重复扫描 `tasks/*.yaml`

只有在以下情况才回退到 `tasks/*.yaml`：
- `TASKS.md` 缺失
- `TASKS.md` 信息明显过期或冲突
- 用户明确要求查看单个 TaskCard 的 DoD / 风险 / 依赖 / 历史
- 需要校验摘要和底层任务卡是否一致

### 3.2 TodoContext 最小结构

```yaml
lastUpdated: <timestamp>
focus:
  id: <task-id>
  title: <title>
  status: <status>
  nextStep: <next-step>
active:
  - id: <task-id>
    title: <title>
    status: <status>
pending:
  - id: <task-id>
    title: <title>
    status: <status>
roleView:
  role: <当前进程>
  focus: <task-id | none>
```

## 4. 任务字段

每个任务至少包含：
- 任务 ID
- 名称
- 类型（系统 / 交付 / 实验 / 运维 / 设计 / 实现 / 测试）
- 所属进程（小青 / plan / architect / coder / env-engineer / debugger / tester）
- 状态（Queued / Ready / In Progress / Waiting / Verifying / Done / Cancelled）
- 优先级（P1-P4）
- 目标结果
- 验收标准
- 阻塞项
- 下一步

## 5. 回复模板规则

### 任务相关回复，必须包含
1. **当前任务**：任务 ID + 名称 + 状态
2. **任务动作**：这条回复推进了什么
3. **若当前任务完成**：追加本进程其他进行中 / 未完成任务

### 5.0 默认进度表展示规则

默认回复顺序：
1. `项目进度表`
2. `当前进行中任务状态`
3. 正文回复

标准展示模板：

```md
## 项目进度表

| 排序 | 项目/任务 | 优先级 | 起始时间 | 结束时间 | 时间进度 | 状态 | 当前进展 | 下一步 |
|---:|---|---|---|---|---:|---|---|---|
| 1 | <文字标题> | <P1/P2/P3/P4> | <startAt> | <endAt / -> | <ratio / -> | <🟩/🟨/🟥/⬛/-> | <一句话> | <一句话> |
| 2 | <文字标题> | <P1/P2/P3/P4> | <startAt> | <endAt / -> | <ratio / -> | <🟩/🟨/🟥/⬛/-> | <一句话> | <一句话> |

## 当前进行中任务状态

| 当前任务 | 执行agent | 状态 | 当前动作 | 当前进展 | 下一步 | 风险/阻塞 |
|---|---|---|---|---|---|---|
| <TASK-ID> / <任务名> | <小青 / plan / architect / coder / env-engineer / debugger / tester> | <状态> | <现在在做什么> | <一句话> | <一句话> | <无 / 具体内容> |

## 本次回复

<正文内容>
```

如果当前没有活跃任务，则写：

```md
| 当前任务 | 执行agent | 状态 | 当前动作 | 当前进展 | 下一步 | 风险/阻塞 |
|---|---|---|---|---|---|---|
| 无 | - | - | - | - | - | - |
```

字段约定：
- `项目/任务` 只使用文字标题
- `状态` 只显示色块，不附加文字说明
- 没有 `deadline` 的任务：`结束时间=-`、`时间进度=-`、`状态=-`
- 无 `deadline` 任务统一排在最后
- 默认只展示 3 到 5 条关键任务

排序规则：
1. 先按 `结束时间` 升序
2. 结束时间相同时，再按优先级排序：`P1 > P2 > P3 > P4`
3. 若仍相同，再按 `起始时间` 升序
4. 没有 `deadline` 的任务统一排在最后
5. 无 `deadline` 任务内部，再按优先级和起始时间排序

颜色计算规则：

```text
时间进度 = (当前时间 - 起始时间) / (结束时间 - 起始时间)
```

- `< 1/3`：`🟩`
- `< 2/3`：`🟨`
- `< 1`：`🟥`
- `>= 1`：`⬛`

如果缺少 `起始时间` 或 `结束时间`，则显示 `时间进度=-`、`状态=-`。

### 5.1 高频待办查看默认路径

对于“看看当前待办”“还有什么没做完”“下一步是什么”这类高频动作，使用以下路径：

1. 直接读取当前会话里的 `TodoContext`
2. 如本次会话刚修改任务状态，先更新 `TASKS.md`，再刷新 `TodoContext`
3. 只有摘要不够时，才下钻读取具体 `tasks/<task-id>.yaml`

目标是把待办查看从“目录扫描动作”改成“启动后内存读取动作”。

### 普通聊天
不强制带任务字段。

## 7. 多任务并行约束

- 默认只允许 **1 个当前聚焦任务**
- 同一进程的活跃任务尽量不超过 3 个
- 如果存在多个未完成任务，回复时优先暴露它们，避免“做完一个就忘了别的”
- 若新任务会打断当前聚焦任务，小青必须先显式说明切换原因

## 8. 与 TOC 的关系

任务系统不是为了记录而记录，而是为了服务 TOC：
- 帮助识别当前主制约
- 暴露库存（等待、阻塞、返工）
- 约束不良多工
- 让每条任务相关回复都保持焦点

## 9. 维护规则

- 新任务进入系统时，先写入 `TASKS.md`
- 任务状态变化时，更新 `TASKS.md`
- 任务完成时，迁移到“最近完成”并暴露其他未完成任务
- 如果发现旧任务长期无人推进，小青要主动标记阻塞或清理
- 每次 `TASKS.md` 更新后，应把当前会话内的 `TodoContext` 视为已失效并立即刷新，避免回答待办时使用旧摘要
