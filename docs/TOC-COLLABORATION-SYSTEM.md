# TOC 协作系统 v1

最后更新：2026-04-08

## 1. 系统目标

通过“小青 + 俊阳 + 专项子进程”的协作，**高质量、按时完成尽量多的高价值任务**。

不是追求看起来很忙，也不是追求并行很多任务，而是追求系统吞吐。

## 2. 全局指标

### Throughput（吞吐）
单位时间内真正完成并被接受的高价值任务数。

### Inventory（库存）
未澄清任务、进行中任务、待验证任务、返工任务、等待决策任务。

### Operating Expense（运营消耗）
俊阳注意力、小青调度成本、子进程重复劳动、验证与切换成本。

## 3. 当前默认主制约

初始默认假设：**主链路决策吞吐**。

主要体现为：
- 需求澄清慢
- 优先级裁决不清
- 完成定义不稳
- 等待俊阳拍板
- 小青未把问题收敛成可决策包

注意：每个周期只允许有 **1 个当前主制约**。

## 4. 系统分层

### 4.1 目标层（俊阳）
负责：
- 定义阶段目标
- 定义价值排序
- 明确截止时间/承诺窗口
- 明确哪些不做

### 4.2 调度与制约管理层（小青）
负责：
- 收敛任务入口
- 维护唯一优先级队列
- 识别当前制约
- 控制 WIP
- 决定何时释放任务
- 决定是否分发给子进程
- 汇总结果和决策包
- 维护复盘与记忆

### 4.3 专业执行层（plan / architect / coder / env-engineer / debugger / tester）
职责：
- 只消费已释放任务
- 在角色边界内输出结构化结果
- 不争抢优先级，不绕过调度层

### 4.4 验证与验收层
由 tester + 小青 + 俊阳组成：
- tester 验技术正确性
- 小青验过程一致性
- 俊阳验业务价值与最终接受

### 4.5 学习与提升层
沉淀：
- 制约证据
- 返工原因
- 模板改进
- 自动化机会
- 角色/流程升级建议

## 5. 核心运行闭环

每轮都按以下顺序运行：

1. 定义目标
2. 识别当前主制约
3. 挖尽制约
4. 迁就制约
5. 提升制约
6. 重新识别制约

### 5.1 挖尽制约
让当前制约只做最关键的事：
- 减少打断
- 减少切换
- 提前准备输入
- 压缩等待

### 5.2 迁就制约
其他所有环节围绕制约安排节奏：
- 不往制约前堆半成品
- 不抢跑不确定任务
- 不制造额外库存

### 5.3 提升制约
当“挖尽 + 迁就”后仍卡住，再做提升动作：
- 新模板
- 默认决策规则
- 自动化
- 更清晰的接口与验收方式
- 新工具或新角色协作方式

## 6. Drum-Buffer-Rope

### Drum（鼓点）
系统节奏由当前主制约决定。

### Buffer（缓冲）
维护三个关键缓冲区：
- Ready Buffer：已澄清、可执行、待释放
- Constraint Buffer：等待当前制约处理的事项
- Verification Buffer：已实现、待验证/待验收

### Rope（绳子）
任务只有在当前制约可消化时才允许释放进系统。

原则：
- 子进程空闲，不等于要放新任务
- 局部闲一点，优于全局库存爆炸

## 7. 任务状态机

统一状态：
- Captured
- Clarified
- Ready
- Released
- Executing
- Verifying
- WaitingDecision
- Blocked
- Rework
- Done

规则：
- 没有验收标准，不进 Ready
- 未被小青释放，不进 Executing
- tester 未过，不进 Done
- 返工必须记录返工来源

## 8. 制约状态机

- Suspected
- Validated
- Exploiting
- Subordinating
- Elevating
- Reassessing

## 9. 优先级规则

### P1
直接缓解当前主制约的任务。

### P2
保护当前主制约不被无效输入淹没的任务。

### P3
局部优化类任务。

### P4
不改善当前吞吐的忙碌型任务，默认延后或砍掉。

优先级判断只问三句：
1. 它是否直接缓解当前制约？
2. 它是否减少返工或等待？
3. 如果现在不做，会不会影响本周主目标？

## 10. WIP 限制

### 总体
- 同时活跃的真正任务 <= 3
- 深度任务 <= 1 到 2

### 分阶段
- 澄清中 <= 2
- 设计中 <= 1
- 编码中 <= 1
- 测试中 <= 1
- 阻塞中 <= 1

## 10.1 任务系统约束

- 任务单一事实来源：`TASKS.md`
- 普通聊天不算任务，不进入任务系统
- 每次任务相关回复必须明确“当前任务”
- 当前任务完成时，必须顺带暴露本进程其他进行中和未完成任务
- 默认只允许 1 个当前聚焦任务，避免不良多工

## 11. 子进程触发规则

### plan
当目标需要拆解、排期、依赖梳理、节奏重排时触发。

### architect
当任务存在边界不清、技术取舍、接口设计、架构审查时触发。

### coder
当任务目标、边界、验收标准已明确，且需要代码实现时触发。

### env-engineer
当制约来自环境、依赖、部署、运行稳定性时触发。

### debugger
当制约来自异常、故障、未知根因时触发。

### tester
当需要质量门禁、验收、回归验证时触发。

## 12. 小青调度 SOP

### 接到任务时
1. 先用一句话定义目标
2. 判断当前主制约候选
3. 优先给出“最佳实践方案 + 决策点 + 关键风险”
4. 再判断是否需要分发子进程
5. 只有目标/验收清晰后才释放执行

### 需要向俊阳汇报时
统一使用 DecisionPacket，避免碎片化打扰。

### 发现阻塞时
按时限升级：
- 快任务阻塞 > 30 分钟
- 中任务阻塞 > 半天
- 重任务阻塞 > 1 天

一旦超时，回到“小青 + plan”重新识别制约。

## 13. 运行节奏

### 每日启动（10-15 分钟）
输出：
- 今日目标（1 到 3 个）
- 当前主制约（只选 1 个）
- WIP 上限
- 今日第一优先流转项

### 每日收尾（10 分钟）
输出：
- 今天完成了什么
- 哪个环节是真制约
- 最大等待/返工是什么
- 明天第一件事是什么

### 每周复盘（30-45 分钟）
输出：
- 本周目标完成度
- 本周主制约
- 是否已挖尽/迁就/提升该制约
- 制约是否转移
- 下周唯一优先改进项

## 14. 第一周最小执行版本

第一周只做四件事：
1. 定义唯一周目标
2. 每天明确唯一主制约
3. 把总 WIP 压到 3 以内
4. 所有任务先补齐验收标准再进实现

这四件事是 v1 的最小控制面。
