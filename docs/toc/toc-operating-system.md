# TOC 持续优化协作系统

版本：v1.0  
适用范围：俊阳 + 小青 + plan / architect / coder / env-engineer / debugger / tester

---

## 1. 系统目标

### 1.1 全局目标
通过“小青 + 俊阳 + 子进程角色”的协作，**高质量、按时地完成尽量多的高价值任务**。

### 1.2 衡量标准
系统统一使用 TOC 三指标：

- **Throughput（吞吐）**：单位时间内，真正完成并验收通过的高价值任务数
- **Inventory（库存）**：所有尚未转化为已验收结果的工作积压
- **Operating Expense（运营消耗）**：推动系统运转所消耗的注意力、协调、返工和验证成本

### 1.3 系统原则
- 追求全局吞吐，不追求局部忙碌
- 未澄清任务不释放
- 未验证任务不算完成
- 同一时间只认一个主制约
- 所有角色围绕主制约协同
- 所有返工必须可追溯

## 2. 默认系统判断

### 2.1 当前默认主制约
在没有证据推翻前，默认优先怀疑当前主制约为：

**主链路决策吞吐，即“俊阳 ↔ 小青”的目标澄清、优先级裁决、验收确认能力。**

### 2.2 运行方式
系统按 Drum-Buffer-Rope 运行：

- **Drum**：当前主制约决定系统节奏
- **Buffer**：保护制约不断料，也不被噪音击穿
- **Rope**：只有制约可消化时，才释放新任务

## 3. 系统架构

### 3.1 目标层
由俊阳定义：

- 当前阶段目标
- 价值排序
- 截止时间
- 范围边界
- 不做什么

### 3.2 调度控制层
由小青负责：

- 统一任务入口
- 创建和维护 TaskCard
- 识别主制约
- 控制 WIP
- 决定任务是否 Released
- 决定调用哪个角色
- 汇总为 DecisionPacket
- 主持复盘

### 3.3 专业执行层
- `plan`：规划、拆解、依赖、节奏
- `architect`：架构、接口、边界、技术权衡
- `coder`：实现
- `env-engineer`：环境与依赖稳定性
- `debugger`：根因定位与故障隔离
- `tester`：验证、验收门、质量结论

### 3.4 验证层
进入 Done 前必须完成：

- 技术正确性验证
- 架构一致性验证
- DoD 验证
- 价值验收确认

### 3.5 学习提升层
沉淀和更新：

- 当前主制约记录
- 返工根因
- 模板与协议缺陷
- 自动化机会
- 提升实验结果

## 4. 核心对象

### 4.1 TaskCard
系统中的唯一任务对象，用于承载：

- 目标
- 优先级
- 状态
- 角色归属
- DoD
- 依赖
- 风险
- 阻塞
- 返工
- 时间戳

### 4.2 ConstraintRecord
系统中的唯一主制约对象，用于承载：

- 当前主制约名称
- 证据
- 队列与等待指标
- 利用策略
- 迁就策略
- 提升候选动作
- 复盘时间

### 4.3 DecisionPacket
所有需要俊阳裁决的事项，统一通过此对象交付，避免碎片打扰。

### 4.4 ExperimentCard
所有提升制约的动作，必须以小实验形式记录、执行、评估。

## 5. 状态机

### 5.1 TaskCard 状态机
```text
Captured -> Clarified -> Ready -> Released -> Executing -> Verifying -> Done
                     \-> Blocked
Executing -> WaitingDecision -> Executing
Verifying -> Rework -> Clarified / Ready
Blocked -> Clarified / Ready
```

### 5.2 状态定义
- `Captured`：已收集，未澄清
- `Clarified`：目标、边界、DoD 已明确
- `Ready`：具备进入系统条件
- `Released`：已被小青放行
- `Executing`：角色处理中
- `WaitingDecision`：等待裁决
- `Blocked`：受依赖、环境或外部条件阻塞
- `Verifying`：验证中
- `Rework`：验证未通过，需要返工
- `Done`：已验收完成

### 5.3 强约束
- 无 DoD，不得进入 `Ready`
- 无 owner，不得进入 `Executing`
- 未经 `Verifying`，不得进入 `Done`
- `Rework` 必须记录 `sourceOfRework`

### 5.4 ConstraintRecord 状态机
```text
Suspected -> Validated -> Exploiting -> Subordinating -> Elevating -> Reassessing
Reassessing -> Validated | Archived
```

## 6. 运行闭环

### 6.1 发现制约
小青每日至少回答一次：

- 哪个队列最长
- 哪个环节等待时间最高
- 哪个环节返工最多
- 哪个环节一堵，全局最伤

默认排查顺序：

1. 主链路决策吞吐
2. 验证吞吐
3. 环境稳定性
4. 实现吞吐
5. 规划拆解质量

### 6.2 利用制约
如果当前主制约是主链路决策吞吐，则必须：

- 非关键事项不升级给俊阳
- 所有待决策事项先由小青收敛成包
- 决策按包处理，不碎片打扰
- 未澄清任务不释放
- 优先保护主链路时间块

### 6.3 迁就制约
各角色必须围绕主制约协同：

- `plan` 输出可执行计划
- `architect` 输出清晰边界与取舍
- `coder` 不抢跑未澄清任务
- `tester` 尽早暴露验收分歧
- `debugger` 优先清掉堵塞主制约的问题
- `env-engineer` 减少低级环境问题打断主链路

### 6.4 提升制约
只有当“利用 + 迁就”都做过后，才允许投入提升动作。

对主链路决策吞吐的提升动作示例：

- 统一 intake 模板
- 统一 DecisionPacket 模板
- 批处理决策窗口
- 固化默认优先级规则
- 固化 DoD 模板
- 自动汇总待决策事项

### 6.5 重新识别
出现以下任一信号，进入 `Reassessing`：

- 当前制约队列明显下降
- 平均等待时间明显改善
- 返工率从当前环节转移
- 另一个环节开始持续积压

规则：**每次只承认一个主制约。**

## 7. 角色职责

### 7.1 俊阳
负责：

- 定义阶段目标
- 决定价值排序
- 处理关键分歧
- 做最终价值验收

不负责：

- 日常调度
- 多角色协调
- 低价值细节裁决

### 7.2 小青
负责：

- 维护优先级
- 管理 TaskCard 生命周期
- 识别并维护主制约
- 控制 WIP
- 决定任务释放
- 选择合适角色
- 汇总为 DecisionPacket
- 做日常与每周复盘

### 7.3 plan
负责：

- 阶段拆解
- 依赖整理
- 里程碑设计
- 风险排序

### 7.4 architect
负责：

- 架构决策
- 接口定义
- 组件边界
- 技术权衡
- 对实现和验证的边界约束

### 7.5 coder
负责：

- 按澄清后任务实现
- 提供自检结果
- 提供待验证点

### 7.6 env-engineer
负责：

- 环境稳定性
- 依赖修复
- 配置结论
- 防复发建议

### 7.7 debugger
负责：

- 根因定位
- 隔离手段
- 修复建议
- 防复发建议

### 7.8 tester
负责：

- 验证执行
- 验收结论
- 阻断问题报告
- 是否允许进入 Done

## 8. 调度规则

### 8.1 WIP 限制
初始建议：

- `Executing` 总数 <= 3
- `WaitingDecision` 总数 <= 2
- `Verifying` 总数 <= 2

### 8.2 超阈值处理
- `Verifying` 超阈值：暂停释放新开发任务
- `WaitingDecision` 超阈值：暂停释放需要新裁决的任务
- `Blocked` 增长：优先清阻塞，不加新任务

### 8.3 任务释放条件
TaskCard 进入 `Released` 前，必须满足：

- 已 `Clarified`
- DoD 明确
- 关键依赖已满足
- 当前 WIP 未超限
- 不会进一步挤压主制约

### 8.4 角色触发条件
#### plan
- 目标复杂
- 依赖多
- 时间线需要设计

#### architect
- 技术方案分叉明显
- 接口边界不清
- 返工根因来自架构模糊

#### coder
必须同时满足：
- 任务已 `Ready`
- DoD 明确
- 无关键待决策项
- 边界足够稳定

#### env-engineer
- Blocked 根因来自环境
- 依赖问题反复出现
- 环境波动影响多个任务

#### debugger
- 出现异常
- 根因未知
- 问题堵住关键流转

#### tester
- 存在可验证交付物
- 即将进入 Done
- 历史返工风险高

## 9. 信息流与接口契约

### 9.1 主链路
```text
俊阳提出目标
-> 小青创建 / 更新 TaskCard
-> 小青澄清并判断是否 Ready
-> 小青识别主制约并决定是否 Released
-> 子进程执行
-> 小青汇总
-> tester / architect 验证
-> 俊阳做必要裁决或最终验收
-> 小青记录结果与复盘
```

### 9.2 小青 -> 子进程
```yaml
taskId:
role:
goal:
scope:
  in:
  out:
definitionOfDone:
knownConstraints:
requiredOutput:
```

### 9.3 子进程 -> 小青
```yaml
taskId:
status:
summary:
deliverables:
risks:
openQuestions:
recommendedNextAction:
decisionNeeded:
```

规则：
- 所有子进程输出必须结构化
- 子进程不得直接争抢俊阳注意力
- 所有待决策必须挂到 DecisionPacket
- 所有阻塞必须挂到 TaskCard

## 10. 指标体系

系统初期只跟踪以下 6 个指标：

- 已验收完成任务数
- 平均 Lead Time
- `WaitingDecision` 队列长度
- `Verifying` 队列长度
- 返工率
- 当前主制约名称

不建议初期做复杂仪表盘。

---

## 11. 复盘机制

### 11.1 日常复盘，小青负责
每日回答：

- 今天真正 Done 了什么
- 哪个环节在排队
- 当前主制约是否成立
- 是否发生返工
- 明天最该保护哪段流转

### 11.2 周复盘，小青主持，俊阳关键裁决
每周回答：

1. 当前主制约是谁
2. 是否已挖尽它
3. 其他环节是否已迁就它
4. 是否值得投入提升动作
5. 制约是否开始转移

---

## 12. 反模式

- 未澄清就派 coder
- 多角色同时追问俊阳
- 用“进行中很多”伪装推进
- 测试长期滞后
- 返工不记原因
- 同时优化多个主问题

---

## 13. 落地顺序

1. 建立 4 个模板
2. 建立状态机与 WIP 规则
3. 只追一个主制约
4. 建立固定复盘节奏

## 14. 一句话定义

**小青负责把整个协作系统变成一个围绕当前主制约运转的受控流，而不是一个谁有空谁接活的并行市场。**
