# TOC 系统触发与执行机制

版本：v1.0  
适用对象：小青  
目标：明确 TOC 系统何时、如何被触发和执行

---

## 1. 执行时机总览

TOC 系统不是持续运行的后台进程，而是由**事件触发** + **定期检查**的组合模式。

### 1.1 触发类型
- **事件触发**：当系统状态发生变化时
- **时间触发**：在特定时间点或间隔

### 1.2 执行层级
- **轻量检查**：快速判断是否需要进一步处理
- **完整流程**：执行全套 TOC 闭环

---

## 2. 事件触发

### 2.1 任务到达时
当出现以下任一情况，触发 **收集 + 澄清** 流程：

1. 俊阳提出新需求
2. 子进程回报新问题
3. 发现环境或验证阻塞
4. 出现返工任务

#### 执行动作
- 创建/更新 TaskCard
- 执行收集判断（是否有价值、是否必须现在做）
- 尝试推进到 Clarified

#### 触发条件代码
```
IF (新需求 OR 新问题 OR 新阻塞 OR 返工)
  THEN 执行 Step 1（收集）和 Step 2（澄清）
```

### 2.2 任务状态变化时
当 TaskCard 状态改变时，触发 **WIP 检查 + 主制约识别**：

1. `Ready` -> `Released`
2. `Executing` -> `Verifying`
3. `Verifying` -> `Done` 或 `Rework`
4. 任何状态 -> `Blocked`

#### 执行动作
- 重新计算各环节 WIP
- 检查是否超阈值
- 快速评估主制约是否需要调整

#### 触发条件代码
```
IF (TaskCard.status 变化)
  THEN 执行 WIP 检查
  IF (WIP 超阈值) THEN 触发相应的暂停/清库存策略
```

### 2.3 需要决策时
当出现需要俊阳裁决的事项，触发 **DecisionPacket 打包**：

1. 影响阶段目标
2. 影响交付时间
3. 影响主制约
4. 影响范围边界
5. 默认规则无法裁决

#### 执行动作
- 形成 DecisionPacket
- 判断是否需要立即升级
- 如不需立即升级，则等待批处理窗口

## 3. 时间触发

### 3.1 心跳触发（主机制）
小青使用 OpenClaw 心跳机制定期执行 TOC 检查。

#### 配置方式
在 `HEARTBEAT.md` 中添加：

```md
## TOC 系统检查

- 检查当前主制约证据
- 计算各环节队列长度
- 更新 ConstraintRecord
- 识别是否需要释放新任务
- 判断是否需要日终复盘
```

#### 建议频率
- **高频**：每 2-4 小时一次（轻量检查）
- **低频**：每日 1-2 次（完整流程）

### 3.2 固定时间触发
在某些固定时间点执行特定动作：

#### 每日上午（09:00-10:00）
- 执行完整收集
- 执行完整澄清
- 识别当日主制约
- 决定当日首次释放

#### 每日下午（16:00-17:00）
- 检查 WIP 状态
- 判断是否需要调整策略
- 准备日终复盘材料

#### 每日结束前（20:00-21:00）
- 执行日终复盘
- 更新所有状态
- 形成明日计划

### 3.3 周复盘触发
每周固定时间执行周复盘：

- 时间：每周五 17:00-18:00
- 输入：本周所有 TaskCard 状态变化
- 输出：下周策略

## 4. 执行流程整合

### 4.1 事件 + 时间组合
系统按以下逻辑运行：

```text
事件触发（快速响应）
  -> 轻量检查
  -> 如需要，触发完整流程
  
时间触发（定期保障）
  -> 完整检查
  -> 系统性调整
```

### 4.2 避免过度触发
为防止系统被频繁触发拖垮，设置以下限制：

- **冷却时间**：相同类型事件 30 分钟内不重复触发完整流程
- **批量处理**：多个相似事件合并处理
- **重要性过滤**：低重要性变化只触发轻量检查

## 5. 执行入口设计

### 5.1 主入口：心跳响应
小青在收到心跳时，执行以下逻辑：

```python
def on_heartbeat():
    # 1. 检查是否有待处理事件
    if has_pending_events():
        handle_events()
    
    # 2. 定期检查
    if is_time_for_check():
        run_toc_check()
    
    # 3. 时间点检查
    if is_daily_morning():
        run_morning_routine()
    if is_daily_evening():
        run_evening_routine()
    if is_weekly_review_time():
        run_weekly_review()
    
    return "HEARTBEAT_OK"
```

### 5.2 事件入口：状态变化监听
当检测到状态变化时：

```python
def on_state_change(task_id, old_status, new_status):
    # 记录变化
    log_state_change(task_id, old_status, new_status)
    
    # 触发相应检查
    if is_wip_related_change(old_status, new_status):
        check_wip_limits()
    
    if is_constraint_related_change(old_status, new_status):
        quick_constraint_check()
    
    # 不立即触发完整流程，等定期检查时处理
```

### 5.3 手动入口：用户请求
当俊阳明确要求时，立即执行：

- 完整系统检查
- 特定环节优化
- 紧急策略调整

## 6. 与现有系统的集成

### 6.1 与 OpenClaw 心跳集成
TOC 系统作为心跳响应的一部分：

```md
# HEARTBEAT.md 配置示例

## 常规检查
- 检查邮件、日历等

## TOC 系统检查（每 4 小时）
- 扫描 TaskCard 目录
- 计算各环节状态
- 更新 ConstraintRecord
- 如需要，调整释放策略

## TOC 日终复盘（每日 20:00）
- 执行日终复盘五问
- 更新明日计划
```

### 6.2 与子进程产出集成
子进程完成时，自动触发相应检查：

- coder 完成 -> 触发验证检查
- tester 完成 -> 触发 Done 检查
- 任何阻塞 -> 触发 Blocked 检查

### 6.3 与文件系统集成
通过监控文件变化触发：

- `tasks/` 目录新增文件 -> 任务到达事件
- TaskCard 文件修改时间变化 -> 状态变化事件
- `constraint-record.yaml` 修改 -> 制约变化事件

## 7. 执行资源管理

### 7.1 计算开销
- **轻量检查**：< 10 秒，可频繁执行
- **完整流程**：1-3 分钟，每日 2-3 次
- **周复盘**：5-10 分钟，每周一次

### 7.2 避免过度执行
- 同一环节 30 分钟内不重复完整检查
- 批量处理相似事件
- 非关键变化只记录不立即处理

### 7.3 故障恢复
如执行失败：

1. 记录失败原因
2. 跳过本次，等待下次触发
3. 如连续失败，降低检查频率
4. 报告俊阳需要人工介入

---

## 8. 一句话总结

**TOC 系统由事件（任务到达/状态变化）触发快速响应，由时间（心跳/固定时间点）触发系统检查，两者结合确保系统既灵敏又不过载。**
