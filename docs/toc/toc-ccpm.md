# TOC 的 CCPM 落地方案

版本：v0.1  
适用范围：俊阳 + 小青 + 子进程协作

## 1. 目标

把现有 TOC 协作系统从“任务流控制”推进到“CCPM 节奏控制”：
- 用**关键链**而不是任务堆积来驱动节奏
- 用**缓冲**而不是逐任务催办来控风险
- 用**任务释放**而不是同时开很多活来保吞吐

## 2. 当前定义

### 2.1 我们这里的关键链
当前不是传统多人甘特图项目，所以关键链定义为：

`目标澄清 -> 方案收敛 -> 实现 -> 验证 -> 验收`

其中任何一段只要长期排队，就可能成为当前主制约。

### 2.2 Drum / Buffer / Rope 对应到 CCPM
- **Drum**：当前关键链中最慢、最稀缺的环节
- **Project Buffer**：放在最终交付前，用来吸收整体不确定性
- **Feeding Buffer**：放在关键链前的依赖接入点，防止侧链拖慢主链
- **Resource Buffer**：在关键角色即将上场前提前准备，避免人到活不到、活到人不在
- **Rope**：只有关键链可消化时才释放新任务

## 3. 四个执行对象

### 3.1 关键链任务
满足以下任一条件就视为关键链任务：
- 直接影响当前阶段目标
- 卡住后续多个任务
- 需要俊阳决策或最终验收
- 是当前主制约上的工作

### 3.2 项目缓冲
先不做复杂百分比燃尽，v0.1 只做三级判断：
- 绿：缓冲安全
- 黄：缓冲开始被吃掉，需要控释放
- 红：缓冲危险，停止加新任务，优先保关键链收口

### 3.3 资源缓冲
对关键角色提前做准备：
- coder 上场前，需求、边界、DoD 必须齐
- tester 上场前，验收口径、样例、检查点必须齐
- 俊阳上场前，DecisionPacket 必须先收敛成包

### 3.4 供给缓冲
如果某项依赖未满足，但后面会阻塞关键链，提前标成 Feeding Buffer 风险，而不是等它真正卡住再说。

## 4. 运行规则

### 4.1 任务释放规则
只有同时满足以下条件才释放到执行：
- 已 Clarified
- DoD 明确
- 不会挤压当前关键链
- 需要的资源缓冲已准备好
- 当前 WIP 未超限

### 4.2 WIP 规则
继续沿用现有上限：
- Executing <= 3
- WaitingDecision <= 2
- Verifying <= 2

但 CCPM 下新增一条：
- **非关键链任务不能抢占关键链资源**

### 4.3 汇报规则
以后默认优先汇报：
1. 当前关键链是什么
2. 当前缓冲颜色
3. 哪个点在吃缓冲
4. 下一步是保链路还是放新任务

## 5. 当前默认判断

基于现有系统，默认先按下面的关键链跑：

`俊阳决策 / 小青澄清 -> architect / coder 执行 -> tester / 俊阳验收`

当前默认主制约优先怀疑：
- 决策吞吐
- 验证吞吐

## 6. 今日起的最小落地动作

### 6.1 小青执行
- 恢复 TOC 主任务运行状态
- 恢复 TOC 心跳检查
- 后续回复增加 CCPM 视角：关键链、缓冲、风险点

### 6.2 第一轮只盯 3 个问题
1. 当前关键链上的任务是谁
2. 当前缓冲是否在被吃掉
3. 现在该不该再放新任务

### 6.3 暂不做的事
- 不先上复杂甘特图
- 不先做精细概率估算
- 不先做多项目算法优化

先把节奏跑起来，再补仪表盘。

## 7. 成功标准

如果接下来几天出现以下变化，说明 CCPM 开始起效：
- 同时开工的任务更少，但 Done 更多
- WaitingDecision 和 Verifying 库存下降
- 任务汇报从“谁在忙”转成“关键链卡在哪”
- 俊阳收到的请求更少，但更集中、更高质量

## 8. 一句话规则

**先保护关键链，再决定要不要释放新任务。**
